WORKPLACE SQL FIELD GUIDE

Write SQL that is safe to share

REQUIREMENTS

Define before querying

Write the decision, result grain, metric formula, population, date window, timezone, status rules, and expected output. “Monthly customers” is incomplete until “customer” and “month” are defined.

Definition of done: expected columns, validation totals, owner, deadline, and known limitations.

SECURITY

Use least privilege

Analysts normally need read access to approved schemas, not global administrator access. Use separate development and production credentials; never place passwords in SQL files or repositories.

-- An administrator runs grants; analysts do not self-grant.
CREATE USER 'report_reader'@'10.%' IDENTIFIED BY 'managed-secret';
GRANT SELECT ON analytics.* TO 'report_reader'@'10.%';
SHOW GRANTS FOR 'report_reader'@'10.%';
SAFE APPLICATIONS

Parameterize input

Application code must send user values as parameters, not concatenate them into SQL text. Parameters reduce injection risk; allow-list validation is still needed for dynamic identifiers such as selected sort columns.

Unsafe: "... WHERE email = '" + email + "'"
Safe:   cursor.execute("... WHERE email = %s", (email,))
CHANGE CONTROL

Make writes reversible

Preview the exact rows, back up when policy requires it, start a transaction on a transactional engine, change narrowly, validate row counts and totals, then COMMIT or ROLLBACK. Test DDL behavior because transaction support varies.

SET OPERATIONS

Stack compatible results

UNION ALL preserves duplicates and is usually the correct choice when stacking independent populations. UNION removes duplicate result rows and adds work. Each branch must return compatible columns in the same order.

Workplace knowledge check and answers
  1. A dashboard asks for active customers. What comes first? Define “active,” time window, timezone, and grain before querying.
  2. Should a reporting account receive ALL PRIVILEGES? No. Grant only the approved operations and schemas.
  3. When should UNION replace UNION ALL? Only when duplicate result rows must be removed by the business definition.
  4. Can parameter placeholders represent a column name? Usually no. Use an allow-list for dynamic identifiers.
  5. Does START TRANSACTION make every DDL change reversible? No. MySQL DDL may cause implicit commits; test behavior and follow change-control policy.

CUMULATIVE ASSESSMENTS

Three gates before the capstone

Work without lesson solutions. A gate passes only when the query runs, its result is validated, and you can explain why the logic is correct.

GATE 1 · AFTER SECTION 4

Beginner analyst

  1. Return Coffee products priced at least 2.000, highest price first.
  2. Calculate total order rows, units, and revenue without confusing their meanings.
  3. Build daily revenue with one row per date.
  4. Explain FROM, WHERE, GROUP BY, SELECT, and ORDER BY.

Pass: 4/4 correct, totals verified, and no SELECT * in final work.

GATE 2 · AFTER SECTION 8

Independent analyst

  1. Build a LEFT JOIN audit preserving every product.
  2. Explain the name-based relationship and redesign it with keys.
  3. Return categories above average product revenue using a CTE.
  4. Validate join row counts and revenue.

Pass: 3/4 fully correct; validation is mandatory.

GATE 3 · AFTER SECTION 13

Advanced analyst

  1. Rank products inside categories and keep the top two.
  2. Build daily revenue, running revenue, and daily change.
  3. Clean staging data safely in a transaction.
  4. Use EXPLAIN evidence to justify or reject an index.

Pass: 3/4 fully correct; safe cleaning and evidence are mandatory.

Reveal grading guidance after attempting

Compare against relevant mini-project solutions. Grade result grain, filter timing, NULL behavior, totals, and explanation. Different SQL is valid when it returns the correct result and preserves business meaning.

PORTFOLIO CAPSTONE · 8–12 HOURS

BeanRoute multi-store sales investigation

Stakeholder brief

The operations director needs a repeatable monthly report. Leaders suspect inconsistent product names, cancelled orders in revenue, and underperforming stores. Turn imperfect transactions into verified findings and recommendations.

Start with capstone_setup.sql. It creates a separate beanroute_capstone database.

Required business questions

  1. What are completed revenue, units, and order count by store and month?
  2. Which products lead revenue within each category?
  3. How do stores compare with company-wide average monthly revenue?
  4. Which customers have not purchased in the latest 30-day period represented?
  5. Where do missing, blank, duplicated, or invalid values threaten analysis?
  6. What changed month over month, and which two findings deserve action?

Required deliverables

  • 01_profile.sql documenting issues before changes
  • 02_clean.sql with reversible cleaning and assumptions
  • 03_analysis.sql answering all six questions
  • 04_reporting_view.sql creating one documented view
  • 05_performance.sql with before/after EXPLAIN evidence
  • README.md covering grain, definitions, findings, limitations, and recommendations
  • One exported result supporting each major finding

Quality assurance

  • State every source and result grain.
  • Exclude cancelled orders; document treatment of pending orders.
  • Reconcile detail and summary revenue.
  • Check join cardinality before aggregates.
  • Handle NULL and blank text deliberately.
  • Use deterministic ranking.
  • Separate evidence from assumptions.
  • Confirm optimized queries return identical results.

TIMED INTERVIEW LAB

Explain while you query

Use the capstone database. Spend two minutes clarifying grain and assumptions, fifteen writing SQL, and three checking and explaining.

1. Top product per store

Return the highest completed-revenue product in each store. Explain ties.

Expected: aggregate store-product revenue, then use ROW_NUMBER for one winner or DENSE_RANK to preserve ties.

2. Customers with no completed order

Return every customer who never completed an order.

Expected: NOT EXISTS, or LEFT JOIN with the status condition in ON and a NULL test.

3. Month-over-month store change

Return monthly completed revenue and percentage change per store.

Expected: monthly aggregation, LAG partitioned by store, and NULLIF around the denominator.

4. Diagnose doubled revenue

A promotions join doubles revenue. Explain the investigation.

Expected: state grains, count before/after joins, test uniqueness, identify many-to-many expansion, and correct the join stage.

5. Optimize a date report

Improve WHERE YEAR(order_date)=2026.

Expected: half-open date range, EXPLAIN, a justified index test, and identical-result verification.

Interview signalInterviewers evaluate SQL, clarifying questions, grain and NULL awareness, validation, readability, and honest treatment of ambiguity.

PUBLISH THE EVIDENCE

Build a reviewable portfolio

  1. Create a repository with numbered deliverables. Never include passwords or private data.
  2. Lead the README with the business problem and two strongest findings, then data, definitions, methods, validation, and limitations.
  3. Include a schema diagram and small result images, not generated database files.
  4. Document clean setup and ask another person to reproduce it.

  • I safely profile and clean data inside a transaction.
  • I read basic EXPLAIN output without unsupported performance claims.
  • My portfolio runs from its documented setup.
  • I completed five interview prompts on time and explained tradeoffs.
  • My resume links to the project and describes evidence accurately.
