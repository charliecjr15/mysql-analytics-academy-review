WHAT CHANGED

The course should teach judgment, not only commands

ANALYST WORKFLOW

Start with the question

Before writing SQL, define the business decision, metric formula, grain, filters, date window, and expected output. Good analysts do not ask the database vague questions.

DATABASE THINKING

Understand the model

Tables, keys, relationships, and constraints explain why a query works. Learners should be able to sketch how orders, products, stores, and customers relate before joining them.

PROOF

Validate every result

A query that runs can still be wrong. Check row counts, join cardinality, NULL behavior, excluded statuses, and reconciliation totals before trusting a result.

CONCEPT COVERAGE

What each major SQL topic is for

DDL

Define structures

CREATE, ALTER, and DROP shape databases and tables. Analysts need enough DDL to understand schemas, create practice tables, and read setup scripts safely.

DML

Change rows carefully

INSERT, UPDATE, and DELETE require previews, transactions, and narrow conditions. In real work, write access is controlled and mistakes can affect downstream reports.

SET OPERATORS

Stack result sets

UNION ALL is usually best when combining compatible populations. Use UNION only when the business rule truly requires duplicate removal.

FUNCTIONS

Prepare values for reporting

String, numeric, date, NULL, and CASE functions turn raw columns into labels, periods, validated measures, and readable report fields.

WINDOWS

Analyze without collapsing rows

Window functions handle rank, running totals, moving comparisons, prior-period values, and percent-of-total calculations while preserving row-level context.

ADVANCED OBJECTS

Know what exists

Views are useful for reporting layers. Temporary tables and CTAS can support staged analysis. Stored procedures, triggers, and partitions are important to recognize, but they are not always part of junior analyst work.

PROJECT LADDER

Use projects to connect the pieces

  1. Beginner report: product, category, and daily sales KPIs from one clean schema.
  2. EDA project: profile a new dataset, document quality issues, clean a copy, and answer business questions.
  3. Reporting layer: create one or more views with documented grain and metric definitions.
  4. Performance note: compare an inefficient filter with a sargable version using EXPLAIN evidence.
  5. Portfolio capstone: combine cleaning, joins, CTEs, windows, QA, and written recommendations in a reproducible repository.
Project ruleA portfolio project is not finished until another person can run it from setup, reproduce the result, and understand the business conclusion.

COMMON FAILURE POINTS

Avoid these traps

MEMORIZATION

Learning syntax without purpose

Every command should answer: what rows, what grain, what metric, and what validation proves it?

JOIN RISK

Trusting joins too early

Always check row counts before and after joins. Duplicated dimension rows or many-to-many joins can silently inflate revenue.

NULLS

Treating missing values as ordinary text

NULL, blank strings, and zero mean different things. Profile them separately and document the business meaning.

DATES

Using vague time filters

Prefer explicit ranges. For timestamp data, use half-open ranges such as >= '2026-06-01' and < '2026-07-01'.

UPDATES

Changing data without a rollback path

Preview with SELECT, change inside a transaction where supported, validate, then commit only when the evidence is correct.

PERFORMANCE

Adding indexes blindly

Indexes speed some reads but cost storage and write work. Use EXPLAIN and identical-result checks before claiming improvement.

PRACTICE LOOP

How to study each lesson

  1. Read the business question and write the expected grain in plain English.
  2. Predict which tables and columns are needed before opening the solution.
  3. Write the query from scratch, then compare with the lesson version.
  4. Run a validation query: row count, total, duplicate check, or NULL check.
  5. Explain the result in one business sentence and one technical sentence.
  6. Change one requirement, such as date range or grouping level, and adapt the SQL.
