Segment 9 — Window Functions

Window functions calculate across related rows without collapsing them into one row per group. They are essential for rankings, running totals, period comparisons, and analytical reports.

These lessons require MySQL 8.0 or newer. Earlier MySQL versions do not support window functions or CTEs.

1. Why window functions matter

GROUP BY reduces many detail rows into summary rows. Sometimes you need a summary and still need to keep each row.

A window function looks across a defined set of rows while preserving the current result's grain.

2. The OVER clause

OVER turns a supported aggregate or ranking function into a window function.

SELECT
    order_id,
    order_date,
    quantity * unit_price AS row_revenue,
    SUM(quantity * unit_price) OVER () AS total_revenue
FROM orders
ORDER BY order_id;

OVER () means the window contains every result row. The output still has one row per order line.

3. GROUP BY versus window functions

This grouped query returns one row:

SELECT SUM(quantity * unit_price) AS total_revenue
FROM orders;

The window query returns every order row plus the same 45.800 total beside each row.

Use GROUP BY when you want one result row per group. Use a window function when you need calculations across rows without losing row-level detail.

4. PARTITION BY

PARTITION BY creates independent windows.

SELECT
    o.order_id,
    p.category,
    o.quantity * o.unit_price AS row_revenue,
    SUM(o.quantity * o.unit_price) OVER (
        PARTITION BY p.category
    ) AS category_revenue
FROM orders AS o
JOIN products AS p ON p.product_id = o.product_id
ORDER BY p.category, o.order_id;

Every Coffee row displays 30.500, while rows from other categories use their own category totals.

5. Running totals

An ordered window can accumulate values over time.

WITH daily_sales AS (
    SELECT
        order_date,
        SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
)
SELECT
    order_date,
    daily_revenue,
    SUM(daily_revenue) OVER (
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_revenue
FROM daily_sales
ORDER BY order_date;

Expected result:

order_date\tdaily_revenue\trunning_revenue
2026-06-20\t13.000\t13.000
2026-06-21\t12.050\t25.050
2026-06-22\t12.500\t37.550
2026-06-23\t8.250\t45.800

6. ROW_NUMBER

ROW_NUMBER assigns a unique sequence based on the window order.

WITH product_sales AS (
    SELECT
        p.product_name,
        SUM(o.quantity * o.unit_price) AS revenue
    FROM products AS p
    JOIN orders AS o ON o.product_id = p.product_id
    GROUP BY p.product_id, p.product_name
)
SELECT
    product_name,
    revenue,
    ROW_NUMBER() OVER (ORDER BY revenue DESC) AS revenue_position
FROM product_sales
ORDER BY revenue_position;

Add a stable tiebreaker such as product_name when deterministic ordering matters.

7. RANK and DENSE_RANK

RANK gives tied values the same rank and leaves gaps afterward. DENSE_RANK gives ties the same rank without gaps.

SELECT
    product_name,
    price,
    RANK() OVER (ORDER BY price DESC) AS price_rank,
    DENSE_RANK() OVER (ORDER BY price DESC) AS dense_price_rank
FROM products
ORDER BY price DESC, product_name;

ROW_NUMBER never ties. Choose the ranking function that matches the business meaning.

8. Ranking inside groups

PARTITION BY restarts ranking for each category.

SELECT
    category,
    product_name,
    price,
    ROW_NUMBER() OVER (
        PARTITION BY category
        ORDER BY price DESC, product_id
    ) AS category_price_position
FROM products
ORDER BY category, category_price_position;

Each category begins again at position 1.

9. Top N per group

Window aliases cannot be filtered in the same SELECT's WHERE clause because WHERE is evaluated before the window calculation. Put the ranking in a CTE, then filter it.

WITH ranked_products AS (
    SELECT
        category,
        product_name,
        price,
        ROW_NUMBER() OVER (
            PARTITION BY category
            ORDER BY price DESC, product_id
        ) AS category_position
    FROM products
)
SELECT category, product_name, price
FROM ranked_products
WHERE category_position <= 2
ORDER BY category, category_position;

This is the standard top-N-per-group pattern.

10. LAG

LAG reads a value from a previous row in the window order.

WITH daily_sales AS (
    SELECT
        order_date,
        SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
)
SELECT
    order_date,
    daily_revenue,
    LAG(daily_revenue) OVER (ORDER BY order_date) AS previous_day_revenue
FROM daily_sales
ORDER BY order_date;

The first row returns NULL because no previous row exists.

11. Period-over-period change

Calculate LAG in one layer, then use the result in the outer query.

WITH daily_sales AS (
    SELECT
        order_date,
        SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
),
with_previous AS (
    SELECT
        order_date,
        daily_revenue,
        LAG(daily_revenue) OVER (ORDER BY order_date) AS previous_revenue
    FROM daily_sales
)
SELECT
    order_date,
    daily_revenue,
    previous_revenue,
    daily_revenue - previous_revenue AS revenue_change
FROM with_previous
ORDER BY order_date;

Do not interpret the first NULL change as zero; there is no earlier period to compare.

12. LEAD

LEAD reads a value from a following row.

WITH daily_sales AS (
    SELECT order_date, SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
)
SELECT
    order_date,
    daily_revenue,
    LEAD(daily_revenue) OVER (ORDER BY order_date) AS next_day_revenue
FROM daily_sales
ORDER BY order_date;

LEAD is useful for comparing a row with what happens next.

13. Moving averages

A frame controls which neighboring rows participate in an ordered window.

WITH daily_sales AS (
    SELECT order_date, SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
)
SELECT
    order_date,
    daily_revenue,
    ROUND(AVG(daily_revenue) OVER (
        ORDER BY order_date
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ), 3) AS three_row_moving_average
FROM daily_sales
ORDER BY order_date;

This is a three-row average, not necessarily a three-calendar-day average if dates are missing.

14. Percentage of total

Divide a row or grouped value by a window total.

WITH product_sales AS (
    SELECT
        p.product_name,
        SUM(o.quantity * o.unit_price) AS revenue
    FROM products AS p
    JOIN orders AS o ON o.product_id = p.product_id
    GROUP BY p.product_id, p.product_name
)
SELECT
    product_name,
    revenue,
    ROUND(100 * revenue / SUM(revenue) OVER (), 2) AS revenue_percent
FROM product_sales
ORDER BY revenue DESC;

The percentages should total approximately 100, allowing for rounding.

15. Window ORDER BY versus result ORDER BY

ORDER BY inside OVER controls the calculation. ORDER BY at the end controls how rows are displayed.

SELECT
    product_name,
    price,
    RANK() OVER (ORDER BY price DESC) AS price_rank
FROM products
ORDER BY product_name;

Ranks are based on descending price even though the final rows are displayed alphabetically.

16. Combining grouping and windows

SQL can aggregate first and then apply a window function to the grouped result.

SELECT
    order_date,
    SUM(quantity * unit_price) AS daily_revenue,
    RANK() OVER (
        ORDER BY SUM(quantity * unit_price) DESC
    ) AS revenue_rank
FROM orders
GROUP BY order_date
ORDER BY revenue_rank;

Expected result:

order_date\tdaily_revenue\trevenue_rank
2026-06-20\t13.000\t1
2026-06-22\t12.500\t2
2026-06-21\t12.050\t3
2026-06-23\t8.250\t4

17. Common mistakes

Common errors include expecting window functions to collapse rows, filtering a window alias in WHERE, omitting a deterministic tiebreaker, confusing calculation order with display order, and assuming a row frame represents uninterrupted calendar time.

Always state the result grain, partition, ordering, and frame in plain language before trusting a window calculation.

18. Final analyst report

Build a daily performance report containing revenue, running revenue, previous revenue, daily change, and revenue rank.

WITH daily_sales AS (
    SELECT
        order_date,
        SUM(quantity * unit_price) AS daily_revenue
    FROM orders
    GROUP BY order_date
),
daily_windows AS (
    SELECT
        order_date,
        daily_revenue,
        SUM(daily_revenue) OVER (
            ORDER BY order_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) AS running_revenue,
        LAG(daily_revenue) OVER (ORDER BY order_date) AS previous_revenue,
        RANK() OVER (ORDER BY daily_revenue DESC) AS revenue_rank
    FROM daily_sales
)
SELECT
    order_date,
    daily_revenue,
    running_revenue,
    previous_revenue,
    daily_revenue - previous_revenue AS daily_change,
    revenue_rank
FROM daily_windows
ORDER BY order_date;

19. Practice Questions

Practice 1: Rank products by revenue

Build product revenue totals, then rank products from highest to lowest revenue.

Answer:

WITH product_sales AS (
    SELECT
        p.product_id,
        p.product_name,
        SUM(o.quantity * o.unit_price) AS revenue
    FROM products AS p
    JOIN orders AS o ON o.product_id = p.product_id
    GROUP BY p.product_id, p.product_name
)
SELECT
    product_name,
    revenue,
    RANK() OVER (ORDER BY revenue DESC) AS revenue_rank
FROM product_sales
ORDER BY revenue_rank, product_name;

Practice 2: Rank prices inside categories

Rank products by price inside each category. Use a deterministic tiebreaker.

Answer:

SELECT
    category,
    product_name,
    price,
    RANK() OVER (
        PARTITION BY category
        ORDER BY price DESC
    ) AS price_rank,
    ROW_NUMBER() OVER (
        PARTITION BY category
        ORDER BY price DESC, product_id
    ) AS stable_position
FROM products
ORDER BY category, stable_position;

Practice 3: Top two products per category

Return the two highest-priced products per category by ranking in a CTE and filtering in the outer query.

Answer:

WITH ranked_products AS (
    SELECT
        category,
        product_name,
        price,
        ROW_NUMBER() OVER (
            PARTITION BY category
            ORDER BY price DESC, product_id
        ) AS category_position
    FROM products
)
SELECT
    category,
    product_name,
    price

Calculate each product's revenue and percentage of total revenue.
